# Favour.ph — Impeccable Design Context

## Design Context

### Users
Two equal personas using a mobile-first web app:

1. **Stressed homeowner** — Something broke (aircon, pipe, electrical). Anxious, needs it fixed fast, trust is everything. Phone in hand, likely already Googled. Scanning fast for credibility signals — verified badge, score, reviews. Needs booking confirmation to feel solid and accountable.

2. **Planned browser** — Weekend maintenance project, comparing options. Relaxed, price-sensitive, wants to read provider bios and service lists properly. May take their time. Both personas occur in equal measure.

**Context:** Mobile web, Philippine users (Batangas City launch), all ages including older adults. Outdoor or home use, varying lighting. Used in moments of either mild stress (urgent fix) or calm comparison (planned work).

### Brand Personality
Three words: **accountable, confident, Filipino**

- Accountable: Every booking has a trail. Providers have public scores. The UI reinforces that this is a serious, trustworthy system — not casual Facebook Messenger coordination.
- Confident: No hedging, no softness. Direct labels, bold numbers, decisive CTAs.
- Filipino: Warm copy (Kuya/Ate prefix for providers), not corporate. Feels local and human, not generic SaaS.

**Emotional goals:** Trust first, then speed. Users should feel safe handing over their home address. The interface must communicate accountability before anything else.

### Aesthetic Direction
**True 50/50: polished shell, dense core.**

- Navigation, transitions, and macro layout feel smooth and consumer-grade (Grab tier).
- Inside cards and forms: information-dense, structured like a field ops tool (hardware store tier).
- Font roles: Manrope 800 for display/headings/names/buttons, Figtree 400–600 for body, JetBrains Mono 700 for all data (prices, codes, distances, scores, ALL CAPS field labels).
- Theme: **Light mode** — used by stressed homeowners outdoors and calm browsers at home. Dark surfaces used intentionally for emphasis only (dark header bar, FavourScore banner, booking confirmed success state).
- Color palette: Primary #0047CC blue (CTAs, active states), near-black #111827 (headers, dark surfaces), verify green #007A33 (trust signals), danger red #D92121 (errors/cancellations), amber #B36B00 (pending states).

**Anti-references:** No glassmorphism, no gradient backgrounds, no floating blobs, no "app store screenshot" generic feel. Nothing that looks like a Dribbble portfolio piece. Feels more like Linear or a Philippine government permit system that got a proper redesign — serious, not pretty.

**Vibe reference:** Grab meets a Philippine hardware store. Clean like Linear, structured like a field ops tool. The one thing someone remembers: the Favour Score banner (dark card, large mono number, credibility-first).

### Design Principles

1. **Trust before delight.** Every screen's primary job is to make users feel safe. Verified badges, scores, and reference codes are visual anchors — never bury them.
2. **Data is the design.** Prices, scores, distances, and timestamps in JetBrains Mono are a deliberate aesthetic choice. The data is the UI — don't decorate around it.
3. **Decisive, never soft.** 52px buttons, 1.5px borders, Manrope 800 everywhere structural. No rounded blobs, no ghost-button-first layouts. The primary CTA is always unambiguous.
4. **Warm copy, cold structure.** The layout is rigid and grid-based. The words are warm and Filipino. "Kuya Marco is available" not "Provider is available."
5. **Mobile-first, thumb-reachable.** Primary CTAs sticky at bottom. Critical actions within thumb zone. Card tap targets ≥ 44px.

### Accessibility
WCAG 2.1 AA: 4.5:1 contrast for normal text, 3:1 for large text and UI components. Touch targets ≥ 44×44px. All interactive elements keyboard-accessible and screen-reader labelled.

### Technical Constraints
- Next.js 14 App Router (TypeScript)
- Tailwind CSS with custom design tokens
- Mobile-first, web only for MVP
- No animations that can't be disabled via `prefers-reduced-motion`
- Server Components for SEO-critical pages (feed, provider profile)
- Client Components only for interactive forms, chat, auth
